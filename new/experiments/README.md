# `experiments/` — what's in this folder

All artifacts here are evaluated on the **75/25/5 partition** of Bugs4Q
(`splits_75_25_5.json`): 32 train, 8 validation, 2 test. Validation is
the headline reporting set. The 2-case test set is too small for
inference and is intentionally not reported on.

This folder contains only the per-case result data and the dataset
description. Statistical-significance tests and synthetic-benchmark
artifacts are deferred to a later evaluation cycle.

---

## Top-level files

### Splits

| File | What it is |
|---|---|
| `splits_75_25_5.json` | The canonical hash-stable Bugs4Q split: train=32, val=8, test=2. Generated by `scripts/resplit.py` with ratios 0.75 / 0.20 / 0.05. The val ID list drives every result file in this folder. |

### Headline GRAP-Q vs Pure-LLM comparison

| File | What it is |
|---|---|
| `combined_results_val.csv` | The primary result file. One row per (case, method) pair: 8 cases × {GRAP-Q, Pure-LLM} = 16 rows. Columns include `lines_f1`, `lines_p`, `lines_r`, `num_edits`, `lines_touched`, `delta_abs_lines`, plus distortion flags (`api_drift_gt40`, `id_jacc_lt60`, `excessive_no_gain`, `ast_parse_fail`, `rationale_autofill`). This is the single source of truth for the per-case GRAP-Q vs Pure-LLM comparison. |

### Cross-method baseline comparison

| File | What it is |
|---|---|
| `baselines_comparison_val.csv` | Per-case comparison of all four methods (GRAP-Q, Pure-LLM, Rule-APR, QChecker) joined by case. One row per case, columns for each method's F1 (Rule-APR and Pure-LLM) or detection signal (QChecker). |
| `baselines_comparison_val.md` | Markdown rendering of the same table with a summary block at the top (mean F1 per method) and a reading guide. Generated by `scripts/compare_baselines.py`. |

### LLM-free baselines

| File | What it is |
|---|---|
| `rule_apr_val.csv` | Per-case results of the deterministic Rule-APR baseline on the 8 val cases. Rule-APR applies seven hand-coded Qiskit-migration rewrites (e.g., `execute(qc, backend=bk)` → `bk.run(transpile(qc, bk))`) and accepts the result only if the rewritten source remains syntactically valid. Columns include `lines_f1`, `num_edits`, and `rules_applied` (which of R1–R7 fired). The file contains only cases where Rule-APR fired (3 of 8), which is why it has fewer rows than the other val files. |
| `qchecker_findings_val.json` | Output of the QChecker-style static analyzer on the 8 val cases. QChecker is a 10-rule AST + regex detector for common quantum-programming defects (deprecated `execute()` API, missing measurement, qubit-index out of range, etc.). Each finding records the rule code (`QC01`–`QC10`), the line, and a brief description. QChecker only flags issues — it does not produce patches. |

### Per-split summary tables

| File | What it is |
|---|---|
| `per_split_val_qchecker.csv` | Per-case QChecker-detection rows aggregated to the val split. Same data as `qchecker_findings_val.json` flattened to one row per case (case, num_findings, rules_hit). |
| `per_split_val_rule_apr.csv` | Per-case Rule-APR rows aggregated to the val split. Same data as `rule_apr_val.csv`. The two files exist for historical reasons (different downstream consumers expect different formats). |

### Prompt ablation (V1–V6)

| File | What it is |
|---|---|
| `ablation/bugs4q_val_ablation.json` | Six-variant prompt-sensitivity ablation on the 8 val cases. Per-row: case, prompt variant, Lines-F1, num edits, wall-time, and guardrail outcome. The summary section reports per-variant means with bootstrap 95% CIs. V1 = production prompt (baseline). V2–V6 = single-element additions or removals (DEPRECATED PATTERNS cheat-sheet, ANTI-PATTERNS prohibition, runtime defect localizer, donor exemplar, minus QUANTUM GUARDRAILS). |

### Dataset analysis

The `dataset_analysis/` folder is a static description of the Bugs4Q
dataset itself (all 42 cases), not a result file. It exists to support
manuscript Section 3 and reviewer questions about dataset composition.

| File | What it is |
|---|---|
| `dataset_analysis/README.md` | Overview: 42 cases, group distribution (Aer, Cirq, StackExchange, etc.), how the split was constructed. |
| `dataset_analysis/all_cases.csv` | One row per case (42 rows). Columns: case ID, group, split label (train/val/test under 75/25/5), buggy/fixed line counts, total lines changed, API drift, identifier Jaccard, QChecker rules hit, Rule-APR F1, Rule-APR rules applied. The most useful single artifact for understanding the dataset. |
| `dataset_analysis/all_cases.json` | Same data as `all_cases.csv` but in richer JSON structure including the full source code of buggy and fixed files. |
| `dataset_analysis/per_split_summary.md` | Aggregate metrics by train/val/test split: mean lines changed, QChecker detection rate, Rule-APR fire rate and mean F1. |
| `dataset_analysis/per_group_summary.md` | Same aggregate metrics but grouped by Bugs4Q source (Aer, Cirq, StackExchange, Terra, etc.) instead of split. |
| `dataset_analysis/bug_patterns_breakdown.md` | Which QChecker rules fire on which cases. Maps each rule (QC01 = MissingMeasurement, QC04 = DeprecatedExecuteAPI, etc.) to the case list. |
| `dataset_analysis/rule_apr_effectiveness.md` | Per-case breakdown of where Rule-APR succeeds, partially fixes, or fails. Shows which deterministic rewrite rules were triggered on each case. |
| `dataset_analysis/case_details/` | One markdown file per case with side-by-side buggy/fixed source, the diff, and a per-case analysis of what the bug is. |

## Headline numbers (descriptive, no formal inference)

Computed on the 8 val cases of `splits_75_25_5.json`:

| Method | Mean Lines-F1 |
|---|---:|
| GRAP-Q | 0.2153 |
| Pure-LLM | 0.1389 |
| Rule-APR | 0.0000 |
| QChecker (detection rate) | 62.5% |

Mean per-case improvement of GRAP-Q over Pure-LLM: **+0.0764**.
Per-case win/loss/tie pattern: **3 / 0 / 5**.

These are descriptive statistics on the val set. Formal hypothesis
testing on this set is deferred; with n=8 and 5 cases at zero F1 for
both methods, any inference would have very limited power.
